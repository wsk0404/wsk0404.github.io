🎯 面试题：什么是 Spring Cloud？它解决了微服务中的哪些问题（含架构剖析 + 面试高分答法）

一、面试速答模板（高分版开场）

Spring Cloud 是一套基于 Spring Boot 的 微服务治理框架，
用于构建分布式系统的完整生态。
它提供了 注册发现、负载均衡、配置中心、熔断限流、网关路由、链路追踪、消息总线 等基础组件。

Spring Cloud 的核心目标是：
✅ 简化微服务的分布式系统开发，
✅ 屏蔽复杂的中间件交互细节，
✅ 提高服务的弹性与可观测性。

二、Spring Cloud 的核心组成模块

功能领域 核心组件（旧版） 主要作用
服务注册与发现 Eureka / Consul / Nacos 让服务之间自动注册和发现
负载均衡 Ribbon / Spring Cloud LoadBalancer 客户端侧负载均衡
服务调用 Feign / OpenFeign 声明式 HTTP 调用
熔断与限流 Hystrix / Resilience4j / Sentinel 防止服务雪崩与过载
配置中心 Spring Cloud Config / Nacos Config 外部化配置统一管理
网关路由 Zuul / Gateway 请求路由与过滤
消息驱动 Spring Cloud Stream / Bus 实现事件驱动与配置广播
链路追踪 Sleuth + Zipkin 追踪分布式请求调用链
安全认证 Spring Cloud Security / OAuth2 统一认证与授权管理

三、Spring Cloud 解决了微服务的哪些核心问题

问题类型 传统微服务痛点 Spring Cloud 解决方案
1️⃣ 服务注册与发现 服务地址动态变化、手动配置困难 Eureka / Nacos 实现自动注册与发现
2️⃣ 服务通信与负载均衡 服务调用难维护，难扩展 Feign + Ribbon 实现声明式调用与负载均衡
3️⃣ 配置管理 多环境配置分散、修改困难 Config / Nacos Config 统一配置中心
4️⃣ 服务容错与降级 服务间依赖导致雪崩 Hystrix / Sentinel 实现熔断、隔离、限流
5️⃣ 路由与网关 多服务对外暴露复杂 Gateway 统一路由入口、鉴权、日志过滤
6️⃣ 分布式日志与链路追踪 请求跨服务难排查问题 Sleuth + Zipkin 实现全链路追踪
7️⃣ 消息通知与配置刷新 配置改动需手动重启服务 Bus 实现配置自动刷新与事件广播
8️⃣ 安全认证 多服务单点登录复杂 OAuth2 + Gateway 实现统一认证授权

四、Spring Cloud 的架构示意图

Spring Cloud 提供了微服务全生命周期管理：

  • 开发期：服务拆分、接口声明；
  • 运行期：注册、路由、熔断、追踪；
  • 运维期：配置中心、监控、动态刷新。

五、Spring Cloud 的设计理念

  1. 约定优于配置（Convention over Configuration）
    → 提供统一标准与默认集成方案。
  2. 轻量化与组件化
    → 各模块松耦合，可按需选择组件。
  3. 与 Spring Boot 无缝集成
    → 一行依赖即可启用分布式能力。
  4. 云原生支持（Cloud Native）
    → 原生适配 Kubernetes、Docker 等环境。

六、Spring Cloud 与 Spring Boot 的关系

对比项 Spring Boot Spring Cloud
定位 微服务框架基础（快速开发） 微服务治理框架（分布式协作）
作用 简化单个服务开发 管理多个服务协作
侧重点 服务内部结构 服务间通信与治理
依赖关系 独立运行 基于 Spring Boot 构建

简而言之：
Spring Boot 解决的是“一个服务怎么跑起来”，
Spring Cloud 解决的是“多个服务如何协同工作”。

七、Spring Cloud 的版本演进

版本代号 主要特性 代表组件
Netflix 版 早期版本，依赖 Netflix OSS Eureka、Ribbon、Hystrix、Zuul
Alibaba 版 新一代云原生微服务 Nacos、Sentinel、Gateway、Seata
Hoxton / 2020+ 与 Spring Boot 2.x / 3.x 兼容 Gateway、OpenFeign、Resilience4j

当前主流选型：Spring Cloud Alibaba（国产化生态完善，Nacos + Sentinel + Seata）

八、面试官高频追问

问题 高分回答
Spring Cloud 与 Dubbo 有什么区别？ Spring Cloud 是一站式微服务治理框架，Dubbo 专注于 RPC 调用。前者偏生态整合，后者偏高性能通信。
Spring Cloud 为什么需要注册中心？ 实现服务动态注册与发现，避免硬编码地址，提高弹性扩展性。
熔断和限流的区别？ 熔断是防止调用失败蔓延，限流是控制请求量防止过载。
为什么推荐使用 Spring Cloud Alibaba？ 支持国产中间件，组件更轻量、兼容性更好、维护活跃。

九、总结（背诵版）

Spring Cloud 是一个微服务治理生态体系，
提供注册发现、配置中心、负载均衡、熔断限流、网关、链路追踪等一站式能力。

它解决了传统分布式系统中：

  • 服务注册与通信困难；
  • 配置与部署复杂；
  • 故障传播与降级缺失；
  • 缺乏统一网关与监控；

使开发者能 专注于业务逻辑，而将分布式基础能力交给 Spring Cloud 来托管。

🎯 面试题：Spring Cloud 服务注册与发现原理（含机制解析 + 核心流程 + 高分答法）

一、面试速答模板（高分版开场）

服务注册与发现（Service Registry & Discovery）是微服务架构的核心机制，
它用于让服务实例能够自动注册自己的网络位置（IP + 端口），
并让其他服务动态发现并调用它们，而无需硬编码地址。

在 Spring Cloud 中，常见的注册中心包括：
Eureka、Nacos、Consul、Zookeeper 等。

核心思想是通过一个 注册中心（Registry Server） 来维护所有服务实例的“实时健康清单”，
实现“服务自动注册、服务自动发现、动态负载均衡”。

二、服务注册与发现的核心流程

整体流程分为三步：

1️⃣ 服务注册（Register）
服务提供者在启动时向注册中心上报自身信息（服务名、IP、端口、元数据等）。

2️⃣ 服务发现（Discovery）
服务消费者从注册中心拉取可用实例列表，实现动态调用。

3️⃣ 心跳检测（Heartbeat）
注册中心周期性检查服务是否存活，及时剔除故障节点。

三、服务注册的原理

以 Eureka / Nacos 为例：

四、服务发现的原理

  • 服务消费者调用前，先从注册中心获取目标服务的地址列表：

  • 注册中心返回所有可用实例信息；

  • 客户端（Ribbon / LoadBalancer）在本地缓存并基于策略（如轮询）选择实例发起请求；

  • Eureka 客户端每隔 30 秒 自动更新注册表（增量同步）。

五、客户端缓存机制（Client-Side Discovery）

Spring Cloud 默认采用 客户端负载均衡模式（Client Side Load Balancing）：

  • 客户端本地维护一份服务列表；
  • 服务调用时，不经过注册中心，而是本地选择目标实例；
  • 优点：调用性能高、注册中心压力小；
  • 缺点：客户端实现复杂，需要定期更新缓存。

对应组件：Ribbon / Spring Cloud LoadBalancer

六、Eureka 的自我保护机制

当网络波动导致大量服务心跳丢失时，Eureka 会触发自我保护模式（Self-Preservation）：

  • 注册中心不会立即下线这些服务；
  • 等待一段时间后再重新评估；
  • 确保分布式环境中“误剔除”减少。

这也是 Eureka 与其他注册中心（如 Consul、Zookeeper）的区别之一：
它更偏向可用性（Availability），而非一致性。

七、Nacos 的改进机制

对比项 Eureka Nacos
一致性模型 AP（高可用） CP + AP 可选
注册协议 HTTP HTTP + gRPC
健康检测 客户端心跳 主动探测 + 客户端上报
动态配置中心 ❌ 无 ✅ 内置
临时实例 / 永久实例 ❌ 无 ✅ 支持
服务权重与分组 ❌ 无 ✅ 支持

Nacos 通过 Push 模式 + gRPC 通道，能实时感知服务变更，
并可结合配置中心，实现“注册 + 配置”一体化。

八、服务注册与发现的核心组件（Spring Cloud 角度）

组件 职责
Eureka Server 注册中心服务端，维护实例列表
Eureka Client 服务端与客户端共同使用，负责注册与拉取
DiscoveryClient 统一抽象接口，屏蔽不同注册中心差异
@EnableEurekaClient 启用 Eureka 客户端功能
@LoadBalanced RestTemplate 启用负载均衡调用

十、面试官常追问

问题 高分回答
服务注册表如何保持一致？ Eureka 使用心跳续约 + 增量同步；Nacos 使用 Raft 协议保持一致性。
注册中心挂了怎么办？ 客户端缓存 + 自我保护机制保证短期可用。
服务如何下线？ 服务关闭时主动发送 DELETE /eureka/apps/{app}/{instance} 请求。
Nacos 与 Eureka 区别？ Nacos 集成配置中心、支持临时实例、Push 模式通知；Eureka 是纯注册中心。

十一、总结（背诵版）

服务注册与发现是微服务架构的基础能力。
通过注册中心，服务可自动注册自身信息并实现动态发现调用。

Spring Cloud 通过 Eureka / Nacos 等组件实现：

  • 服务注册（上报元数据）；
  • 服务发现（动态拉取实例）；
  • 心跳检测（保持健康状态）；
  • 自我保护（防止误剔除）。

Eureka 偏向 高可用（AP），
Nacos 支持 一致性与高可用切换（CP + AP），
是当前微服务架构的主流选择。

🎯 面试题：Spring Cloud 负载均衡（含机制原理 + 组件演进 + 答题模板）

一、面试速答模板（高分版开场）

在 Spring Cloud 微服务架构中，负载均衡（Load Balancing） 是指将客户端的请求按照一定算法分发到多个服务实例上，
以实现 高可用、高并发与流量均衡 的目标。

Spring Cloud 的负载均衡经历了两个主要阶段：

  • ✅ Ribbon（Spring Cloud Netflix 早期版本）
  • ✅ Spring Cloud LoadBalancer（官方替代 Ribbon 的新组件）

两者都属于 客户端负载均衡（Client-Side Load Balancing） 模式，
即：在客户端侧完成服务选择与请求分发，而非依赖中心代理。

二、负载均衡的两种模式

模式类型 说明 特点
客户端负载均衡 客户端从注册中心拉取所有可用实例，自行选择目标服务调用。 性能高、无中心瓶颈（Spring Cloud 默认）
服务端负载均衡 请求先发到负载均衡代理（如 Nginx），由代理转发请求。 简单易维护，但存在单点压力

Spring Cloud 属于第一种：Client-Side 模式。

三、Ribbon 工作原理

Ribbon 是 Netflix 提供的一个客户端负载均衡器，
它通过维护一份 服务实例列表，并基于策略算法从中选择一个服务进行调用。

🔹 核心流程

1️⃣ 服务消费者从注册中心（Eureka/Nacos）拉取实例列表；
2️⃣ Ribbon 将列表缓存到本地；
3️⃣ 每次调用时，根据负载均衡策略选择目标实例；
4️⃣ 通过 RestTemplate 或 Feign 发起 HTTP 请求。

策略类 策略说明

八、面试官常追问问题

面试题 高分回答
Ribbon 和 Nginx 有什么区别？ Ribbon 是客户端负载均衡；Nginx 是服务端代理负载均衡。
Ribbon 如何保证高可用？ 本地缓存服务列表 + 定期刷新 + 重试机制。
Spring Cloud 为什么弃用 Ribbon？ Ribbon 停止维护，不支持响应式；新组件 LoadBalancer 更轻量、更灵活。
负载均衡策略可以自定义吗？ 可以，通过实现 ReactorServiceInstanceLoadBalancer。
Feign 是如何实现负载均衡的？ Feign 底层集成 Ribbon 或 LoadBalancer 实现实例选择。

九、总结（背诵版）

Spring Cloud 的负载均衡机制属于 客户端负载均衡模式，
服务调用方从注册中心拉取实例列表，在本地根据策略选取目标服务。

早期通过 Ribbon 实现，提供多种策略（轮询、随机、权重等）；
后期由 Spring Cloud LoadBalancer 替代，支持异步调用与可插拔扩展。

✅ Ribbon：老牌组件，基于同步调用；
✅ LoadBalancer：新一代组件，响应式、轻量、高性能。

在分布式微服务架构中，负载均衡是实现高可用与高性能服务治理的关键基础。

🎯 面试题：Spring Cloud Gateway 是什么（含原理机制 + 架构详解 + 面试追问）

一、面试速答模板（高分版开场）

Spring Cloud Gateway 是 Spring 官方推出的 第二代 API 网关框架，
取代了早期的 Netflix Zuul 1.x，基于 Spring Boot 2.x 与 Spring WebFlux（响应式编程模型） 构建。

它是微服务架构的统一流量入口，用于请求路由、过滤、限流、鉴权、监控等跨服务功能，
支撑高并发、低延迟的网关转发场景。

一句话概括：
Spring Cloud Gateway 是基于 WebFlux 的 高性能、响应式网关组件，
用于构建微服务系统的 统一 API 访问层。

二、Spring Cloud Gateway 的主要功能

功能模块 说明
路由（Route） 决定请求被转发到哪个目标服务。
断言（Predicate） 匹配请求条件，如路径、Header、参数等。
过滤器（Filter） 在请求前或响应后执行逻辑，如鉴权、日志、限流等。
服务发现（Discovery） 自动从注册中心（如 Nacos/Eureka）拉取服务列表，实现动态路由。
负载均衡（LoadBalance） 支持 lb://service-name 形式的自动负载均衡转发。
统一安全控制 可结合 OAuth2、JWT 实现认证与鉴权。
限流熔断支持 集成 Resilience4j、Redis 实现限流、熔断与降级。

三、核心架构原理

请求处理流程：
1️⃣ 客户端发送请求到 Gateway；
2️⃣ 路由系统根据 断言（Predicate） 判断应由哪个路由处理；
3️⃣ 经过一系列 过滤器（Filter） 执行前置或后置逻辑；
4️⃣ 最终将请求转发至目标微服务；
5️⃣ 响应返回时再次经过后置过滤器，可进行日志记录或响应加工。

五、Spring Cloud Gateway 的优势

优势点 说明
✅ 高性能非阻塞 基于 Reactor 与 Netty，支持高并发响应式编程。
✅ 无缝集成 Spring 生态 与 Spring Boot、Spring Cloud 无缝兼容。
✅ 动态路由与服务发现 自动感知注册中心变化。
✅ 可扩展性强 可自定义 Filter、Predicate、限流、认证模块。
✅ 支持微服务治理 集成熔断、限流、灰度发布等能力。

六、Gateway 与 Zuul 对比（面试高频考点）

对比项 Spring Cloud Gateway Zuul 1.x
架构模型 WebFlux（响应式） Servlet（阻塞）
性能 高性能、非阻塞 I/O 较低，阻塞式 I/O
扩展性 高，Filter + Predicate 模型 中等，仅 Filter
动态路由 原生支持 需手动扩展
推荐程度 ✅ 官方推荐 ❌ 已过时

七、典型使用场景

场景 功能
🔹 微服务统一网关 提供统一访问入口与流量控制
🔹 鉴权与认证中心 实现 JWT 或 OAuth2 认证
🔹 日志与监控采集 在 Filter 中记录请求数据
🔹 限流与熔断保护 防止接口过载或雪崩

🎯 面试题：分布式事务中的 2PC 与 3PC（含原理机制 + 流程图解 + 面试延伸）

一、面试速答模板（高分版开场）

在分布式系统中，事务需要跨多个节点保持一致性。
两阶段提交（2PC, Two-Phase Commit） 和 三阶段提交（3PC, Three-Phase Commit） 是早期经典的分布式事务一致性协议。

  • 2PC：协调者通过“准备阶段 + 提交阶段”确保参与者一致执行；
  • 3PC：在 2PC 基础上增加“预提交阶段”，进一步降低阻塞风险并提高容错性。

二、2PC（Two-Phase Commit）原理详解

2PC 是分布式事务中最经典的实现协议之一，由 协调者（Coordinator） 与 参与者（Participant） 两类节点共同完成。

阶段一：准备阶段（Prepare Phase）

  1. 协调者发送 prepare 请求 给所有参与者；
  2. 参与者执行事务操作但不提交，写入本地日志；
  3. 参与者向协调者返回响应：
    • 若执行成功 → 返回 “Yes（同意提交）”；
    • 若执行失败 → 返回 “No（拒绝提交）”。

阶段二：提交阶段（Commit Phase）

  • 若所有参与者都返回 “Yes”：
    • 协调者发送 commit 指令；
    • 各参与者正式提交事务；
  • 若任意一个返回 “No”：
    • 协调者发送 rollback 指令；
    • 所有参与者回滚事务。

示意图：

优点：

  • 实现简单；
  • 能保证强一致性（ACID）。

缺点：

  • 阻塞问题：若协调者宕机，参与者会陷入等待状态；
  • 单点问题：协调者故障会导致系统无法决策；
  • 无法容忍网络分区：节点间通信异常时无法达成一致。

三、3PC（Three-Phase Commit）机制原理

3PC 是对 2PC 的改进协议，引入了一个额外的阶段——预提交阶段（Pre-Commit Phase），让系统在出现异常时能更灵活地处理事务。

阶段一：CanCommit 阶段

  1. 协调者询问各参与者：“是否可以提交？”
  2. 参与者检查事务可行性，返回 Yes/No。

阶段二：PreCommit 阶段（预提交）

  • 如果所有参与者返回 “Yes”，则协调者发出 PreCommit 通知：
    1. 参与者执行事务操作；
    2. 写入 预提交日志；
    3. 并立即返回 “ACK（确认）” 给协调者。

阶段三：DoCommit 阶段（正式提交）

  • 若协调者收到所有 ACK：
    • 发出 DoCommit 指令，所有参与者正式提交；
  • 若超时或部分失败：
    • 发出 Abort 指令，各参与者回滚事务。

改进点：

  • 3PC 在每个阶段都设置了超时机制；
  • 允许参与者在超时后 自主提交或回滚，减少阻塞；
  • 相比 2PC，具备 更好的容错性 与 可恢复性。

优缺点对比：

对比项 2PC 3PC
阶段数量 2 3
阻塞问题 存在 通过超时机制部分缓解
一致性保障 强一致性（但易阻塞） 一致性稍弱，但可用性更高
容错能力 低（协调者挂则卡死） 较高（参与者可超时自决）
实现复杂度 较低 较高

四、2PC 与 3PC 在实际系统中的应用

系统类型 典型实现方案
传统分布式数据库 XA 协议（基于 2PC）
消息事务系统 RocketMQ 事务消息（2PC思想）
微服务事务 TCC、Saga、可靠消息最终一致性等
分布式协调服务 Zookeeper、Etcd（3PC思想演化）

TCC 与 2PC 的关系：

  • TCC（Try-Confirm-Cancel）可视为 2PC 的业务层实现；
  • Try 对应 Prepare；
  • Confirm 对应 Commit；
  • Cancel 对应 Rollback；
  • 区别在于：TCC 由业务方控制资源，粒度更细、可编程。

五、面试官常追问问题

问题 高分回答
2PC 的最大问题是什么？ 协调者单点故障导致参与者阻塞，系统无法自动恢复。
3PC 如何优化 2PC？ 引入预提交阶段与超时机制，减少阻塞、提升可用性。
3PC 是否绝对安全？ 否，网络分区下仍可能造成数据不一致（CAP 冲突）。
实际系统中还会用 3PC 吗？ 很少直接使用，更多采用基于其思想的改进协议（如 Paxos、Raft、TCC）。
你在项目中如何实现分布式事务？ 结合业务特征，使用可靠消息 + 本地事务 或 TCC 机制实现最终一致性。

六、总结（背诵版）

  • 2PC（两阶段提交）：准备 + 提交；
    • 优点：实现简单，保证强一致；
    • 缺点：阻塞严重、协调者单点。
  • 3PC（三阶段提交）：CanCommit + PreCommit + DoCommit；
    • 优点：引入超时机制，减少阻塞；
    • 缺点：一致性略弱，复杂度高。

实际分布式系统中通常不直接使用 2PC/3PC，而是采用基于其思想的：

  • TCC（业务可控补偿事务）
  • Saga（长事务补偿）
  • 消息事务（最终一致性）

✅ 核心记忆：
2PC 保证一致但易阻塞；3PC 提升可用但弱一致；现代系统以最终一致性为主流。

🎯 面试题：Spring Cloud Seata 是什么（含分布式事务原理 + 模式详解 + 实战场景）

一、面试速答模板（高分版开场）

Spring Cloud Seata 是阿里巴巴开源的 分布式事务解决方案，
用于解决微服务架构下 跨服务、多数据源操作时的数据一致性问题。

它通过 TC（事务协调者）、TM（事务管理者） 和 RM（资源管理者） 的协调机制，
实现全局事务的开启、提交与回滚，确保在分布式环境下 ACID 特性仍然部分成立。

一句话概括：
👉 Seata 是一个高性能的分布式事务协调框架，用来在微服务系统中保持多服务间的数据一致性。

二、为什么需要 Seata？

在微服务架构中，业务通常拆分为多个独立服务，如：

  • 订单服务（Order）
  • 库存服务（Stock）
  • 支付服务（Pay）

当一个业务操作需要同时调用多个服务时，如果中途出现异常（如库存扣减失败），就会导致数据不一致问题。

传统的本地事务（如 MySQL 事务）只能保证单库内一致性，无法解决跨服务事务。

💡 Seata 正是为了解决这种“跨服务分布式事务”问题而生。

三、Seata 的三大核心角色

组件 全称 职责
TC（Transaction Coordinator） 事务协调者 维护全局事务的状态，协调分支事务的提交与回滚。
TM（Transaction Manager） 事务管理者 负责开启、提交、回滚全局事务。
RM（Resource Manager） 资源管理者 管理分支事务的本地资源（数据库连接等），并向 TC 注册。

四、Seata 的工作原理

1️⃣ TM 开启全局事务

  • 通过 @GlobalTransactional 注解标记业务方法；
  • 向 TC 注册全局事务，获取唯一的全局事务 ID（XID）。

2️⃣ RM 注册分支事务

  • 每个参与的微服务（数据库操作）向 TC 注册分支事务；
  • 记录事务的“前镜像”和“后镜像”。

3️⃣ 执行业务逻辑

  • 各 RM 执行本地 SQL 操作，但不提交，只保留在本地。

4️⃣ TM 提交或回滚

  • 如果所有分支执行成功 → TC 通知各 RM 提交；
  • 若任何分支失败 → TC 通知所有 RM 回滚操作，恢复到前镜像状态。

五、Seata 支持的事务模式

模式 简述 适用场景

XA 模式 基于 XA 协议的分布式两阶段提交 强一致性、高可靠但性能较低

🎯 面试题：分布式事务 TCC 模式详解（含执行流程 + 业务示例 + 面试扩展）

一、面试速答模板（高分版开场）

TCC（Try-Confirm-Cancel） 模式是分布式事务中一种“业务层面的补偿型事务”方案，
它通过将每个操作拆分为 三个阶段 来保证最终一致性：

  • Try（尝试）：预留资源；
  • Confirm（确认）：正式提交；
  • Cancel（取消）：回滚补偿。

TCC 模式由业务方控制资源与回滚逻辑，属于一种 “业务可编程事务”，
能在保持高性能的同时，实现 柔性事务（最终一致性）。

二、TCC 的三阶段机制详解

TCC 模式并不依赖数据库层事务，而是由应用服务显式实现事务控制逻辑。

✅ 阶段一：Try（预留资源）

  • 作用：检查业务可行性并预留资源；
  • 示例：冻结账户余额、锁定库存；
  • 设计要求：
    • Try 操作要幂等；
    • 不可对外产生副作用；
    • 必须确保资源“可确认或可回滚”。

✅ 阶段二：Confirm（确认提交）

  • 作用：在所有子事务的 Try 都成功后，执行业务的正式提交；
  • 示例：扣减账户余额、扣除库存；
  • 特点：
    • Confirm 必须保证幂等；
    • Confirm 操作通常不再进行任何校验；
    • 一旦执行，事务进入最终完成状态。

✅ 阶段三：Cancel（取消补偿）

  • 作用：当任意 Try 失败或部分 Confirm 失败时，执行补偿操作；
  • 示例：解冻余额、释放库存；
  • 特点：
    • Cancel 必须保证幂等；
    • Cancel 逻辑与 Try 逻辑相反；
    • 若 Try 未执行成功，则 Cancel 应该能识别并忽略。

三、TCC 执行流程示意图

👉 若任一 Try 失败，事务管理器调用各服务的 Cancel 方法回滚补偿。

关键点 说明

2PC 强一致性 低 中 分布式数据库
TCC 最终一致性 高 高 金融、电商、支付系统

七、TCC 的常见问题与解决方案

九、面试官常追问问题

问题 高分回答
TCC 与 2PC 的区别？ 2PC 是数据库层协议，TCC 是业务层补偿机制；TCC 更灵活但一致性较弱。
TCC 如何保证幂等？ 通过全局事务ID（XID）+ 本地幂等表，防止重复执行。
如何处理 Confirm 阶段失败？ 使用重试机制或补偿任务，确保最终一致性。
TCC 与 Saga 的区别？ Saga 用反向事务补偿，TCC 用显式 Cancel 接口；TCC 实时性更高。
哪些业务适合用 TCC？ 支付、转账、扣库存、积分发放等可控资源型事务。

十、总结（背诵版）

TCC 模式核心：Try + Confirm + Cancel 三阶段补偿机制。

  • Try：预留资源；
  • Confirm：提交事务；
  • Cancel：回滚资源。

特点：

  • 由业务方实现，粒度细，可控性高；
  • 具备幂等性、空回滚、防悬挂机制；
  • 实现最终一致性，性能优于 2PC。

✅ 口诀记忆：
“Try 试探锁资源，Confirm 确认定乾坤，Cancel 回滚保一致。”

优化亮点：

  • 请求拦截：防止用户直接打数据库；
  • 库存预扣减：Redis原子操作；
  • 异步下单：削峰填谷；
  • 超卖控制：Lua 脚本 + 分布式锁。

✅ 案例 2：热点缓存系统

优化亮点：

  • 热点数据全量缓存；
  • 缓存预热、更新、穿透、击穿防护；
  • 缓存一致性通过异步回写、消息同步实现。

六、高并发系统的常见瓶颈

瓶颈层 问题表现 解决手段
前端层 请求洪峰 CDN 缓存 + 限流
应用层 CPU/线程池爆满 异步化、降级、线程池隔离
缓存层 Redis 过载 热点分片、Cluster 模式
数据库层 IO阻塞、锁冲突 分库分表、读写分离、索引优化
消息队列 积压严重 多消费者并行、分区消费

七、高并发常见问题及解决方案

问题 解决方案
缓存穿透 使用布隆过滤器
缓存击穿 加互斥锁或逻辑过期
缓存雪崩 设置随机过期时间 + 限流降级
超卖问题 Redis Lua 原子扣减
重复下单 使用 Redis 分布式锁或 token 验证
消息丢失 MQ 持久化 + 确认机制
系统雪崩 限流、熔断、隔离、降级

八、面试官高频追问

面试问题 高分回答
高并发下如何防止超卖？ 使用 Redis 原子扣减或 Lua 脚本实现库存控制。
如何实现系统限流？ 可以用令牌桶算法（Guava/Sentinel），或 Nginx 配置 limit_req_zone。
如何防止缓存雪崩？ 给缓存设置随机过期时间 + 预热机制 + 多级缓存。
Redis 崩溃后系统如何保证可用？ 启用主从复制 + 哨兵机制 + 持久化（AOF/RDB）。

九、总结（背诵版）

高并发的核心目标是：高性能、高可用、高扩展性。

解决方案遵循三大核心思路：

  • 分流（负载均衡）
  • 削峰（异步消息队列）
  • 提速（缓存 + 分库分表）

常见优化策略包括：缓存、限流、降级、异步、分布式锁、分库分表。
高并发系统设计的本质是：让请求更快地被处理、让失败更快地被隔离、让系统更容易地扩容。

👉 一句话总结：

高并发系统设计 = 缓存 + 异步 + 限流 + 分布式架构。
