🎯 面试题：请你详细说一下 SpringMVC 的工作流程

一、面试官考察点

面试官通过此题主要考察候选人是否：

  • 理解 SpringMVC 的整体架构和请求生命周期；
  • 能清楚说出各组件的职责（DispatcherServlet、HandlerMapping、HandlerAdapter、ViewResolver等）；
  • 能讲明白“请求是如何被定位、执行、渲染”的；
  • 能从源码或调用链角度分析底层执行逻辑。

二、30 秒面试速答版（高分简答）

SpringMVC 是基于 前端控制器（DispatcherServlet） 模式实现的 Web MVC 框架。
当请求到达时，DispatcherServlet 作为核心调度器，按以下步骤处理：

1️⃣ 用户发送请求到前端控制器 DispatcherServlet；
2️⃣ DispatcherServlet 查询 HandlerMapping 寻找匹配的 Handler（控制器方法）；
3️⃣ 通过 HandlerAdapter 调用对应的 Controller 方法；
4️⃣ Controller 执行业务逻辑后返回 ModelAndView；
5️⃣ DispatcherServlet 调用 ViewResolver 解析视图；
6️⃣ 渲染视图并返回响应给客户端。

这一系列流程体现了典型的 “请求分发 - 业务处理 - 响应渲染” MVC 三层架构思想。

八、常见追问与高分回答

追问 高分回答
❓ DispatcherServlet 是单例吗？ 是的，它是单例 Bean，在应用启动时初始化，只处理请求调度逻辑，不保存状态，线程安全。
❓ Controller 是线程安全的吗？ 默认是单例，必须保证方法无共享可变状态，否则需使用局部变量或 @Scope("prototype")。
❓ SpringMVC 如何返回 JSON？ 由 MappingJackson2HttpMessageConverter 将对象序列化为 JSON，响应头为 application/json。
❓ @RestController 和 @Controller 区别？ @RestController = @Controller + @ResponseBody，自动将返回值转为 JSON。
❓ SpringMVC 和 Servlet 区别？ Servlet 处理请求细节，SpringMVC 基于 DispatcherServlet 实现 MVC 架构，解耦 Controller、View、Model。

🔥 十、一分钟总结（可背诵）

SpringMVC 的核心是 DispatcherServlet 前端控制器。
当请求到达时，它会依次：
1️⃣ 调用 HandlerMapping 定位 Controller；
2️⃣ 通过 HandlerAdapter 调用方法；
3️⃣ 返回 ModelAndView；
4️⃣ 调用 ViewResolver 渲染视图；
5️⃣ 最终将结果返回给客户端。

整个流程体现了典型的 MVC 模式：
Controller 控制流程，Model 管理数据，View 渲染展示。

🎯 面试题：SpringMVC 常用注解有哪些？它们的作用是什么？

一、面试官考察点

通过此题考察开发者是否熟悉 SpringMVC 的核心使用方式，包括：

  • 控制层类与方法的声明；
  • 请求映射与参数绑定；
  • 返回值与响应体处理；
  • 数据校验、异常处理等；
  • 是否理解注解的底层原理与组合逻辑。

二、30 秒速答版（面试简答）

SpringMVC 中常用注解分为五大类：

1️⃣ 控制层类相关注解：@Controller、@RestController
2️⃣ 请求映射相关注解：@RequestMapping、@GetMapping、@PostMapping 等
3️⃣ 参数绑定注解：@RequestParam、@PathVariable、@RequestBody、@ModelAttribute
4️⃣ 响应结果处理注解：@ResponseBody、@ResponseStatus
5️⃣ 其他扩展注解：@ControllerAdvice、@ExceptionHandler、@InitBinder、@CrossOrigin

这些注解共同构成了 SpringMVC 的控制层核心机制。

四、核心底层原理分析

1️⃣ 注解解析机制

Spring 在启动时会通过：

  • RequestMappingHandlerMapping 解析控制器中的 @RequestMapping；
  • RequestMappingHandlerAdapter 负责执行对应的方法；
  • HandlerMethodArgumentResolver 处理入参注解；
  • HandlerMethodReturnValueHandler 处理出参注解（如 @ResponseBody）。

2️⃣ 底层本质：

这些注解大多基于 反射 + 元注解机制 + 策略模式 实现。
例如：

  • @GetMapping 是 @RequestMapping(method = RequestMethod.GET) 的快捷注解；
  • @RestController 是 @Controller 和 @ResponseBody 的组合注解；
  • @ControllerAdvice 是一种特殊的 @Component，用于全局增强。

五、常见面试追问与高分回答

追问 高分回答
❓ @RestController 和 @Controller 区别？ @RestController = @Controller + @ResponseBody，前者直接返回 JSON，不走视图解析。
❓ @RequestBody 和 @RequestParam 区别？ 前者解析请求体 JSON，后者解析 URL 参数。
❓ @ModelAttribute 和 @RequestBody 的区别？ @ModelAttribute 用于表单提交（x-www-form-urlencoded），@RequestBody 用于 JSON。
❓ @ControllerAdvice 能作用于 @RestController 吗？ 可以，默认作用于所有 Controller，可通过 basePackages 限制范围。
❓ @InitBinder 有什么用？ 用于自定义数据绑定，比如将字符串转日期或注册自定义编辑器。

六、常见陷阱与注意事项

场景 问题 解决方案
@RequestBody 参数为 null 请求头未设置 Content-Type: application/json 正确设置 Content-Type
@PathVariable 不匹配 路径占位符与变量名不一致 指定参数名：@PathVariable("id")
@ResponseBody 返回中文乱码 响应编码未配置 设置 produces="application/json;charset=UTF-8"
全局异常未生效 未加 @ControllerAdvice 或包扫描不包含 检查包路径与注解

七、加分扩展话题（高级面试方向）

方向 内容
注解元模型 所有注解都继承自 @RequestMapping 元注解体系
参数解析机制 基于 HandlerMethodArgumentResolver 可自定义注解参数解析器
返回值处理机制 HandlerMethodReturnValueHandler 负责对象序列化
定制化扩展 可自定义注解实现统一权限校验或接口日志

🔥 八、一分钟高分总结（可背诵）

SpringMVC 提供了一套完整的注解体系来简化 Web 层开发：

  • @Controller、@RestController：定义控制层；
  • @RequestMapping 系列：定义请求路由；
  • @RequestParam、@PathVariable、@RequestBody：绑定请求参数；
  • @ResponseBody、@ResponseStatus：定义响应行为；
  • @ControllerAdvice、@ExceptionHandler：实现全局异常处理。

底层通过反射机制和 HandlerMapping / HandlerAdapter 组件解析执行，
使整个 MVC 流程完全基于注解驱动，实现了高内聚、低耦合的控制层架构。

🎯 面试题：@RestController 和 @Controller 有什么区别？

一、面试官考察点

  1. 是否理解 Spring MVC 的控制层处理机制；
  2. 是否清楚 视图解析（ViewResolver） 与 消息转换（HttpMessageConverter） 的差异；
  3. 是否了解 SpringBoot 中常见注解的组合原理；
  4. 能否结合实际开发场景说明选择原因（如 WMS 系统接口开发）。

二、核心结论（30 秒快速回答）

@Controller 用于返回 视图页面，一般配合 前后端不分离项目 使用；

@RestController 是 @Controller + @ResponseBody 的组合注解，返回的是 JSON/XML 数据，常用于 前后端分离或接口型系统（如微服务、API）。

三、详细解析

1️⃣ @Controller 的作用

  • 用于标识一个类是 Spring MVC 控制器；
  • 通常用于返回页面（视图），通过 视图解析器(ViewResolver) 渲染 JSP、Thymeleaf、Freemarker 等模板。

  3. 结合 Model 数据渲染 HTML 页面；

2️⃣ @RestController 的作用

  • 是一个组合注解，等价于：

  • 表示该类的所有方法默认返回 JSON 或 XML 数据，而不是视图页面；

  • 常用于 RESTful API 开发（前后端分离、移动端接口）。

四、底层原理分析

对比项 @Controller @RestController
是否返回视图 ✅ 是（交由视图解析器处理） ❌ 否（直接返回数据）

典型应用 MVC 模板页面 RESTful 接口、前后端分离
底层支持 ViewResolver HttpMessageConverter
常用场景 JSP、Thymeleaf JSON API、微服务

八、常见面试追问

面试官提问 高分回答
@RestController 和 @ResponseBody 有什么区别？ @RestController 是 @Controller + @ResponseBody 的组合，作用于类级别；@ResponseBody 可以单独用于方法级别。
如何在 @Controller 中返回 JSON？ 在方法上添加 @ResponseBody 注解。
Spring MVC 如何将对象转为 JSON？ 通过 HttpMessageConverter（如 Jackson 实现）自动序列化。
Controller 层返回结果如何统一封装？ 使用 ResponseEntity 或自定义响应封装类（如 Result<T>）。

九、性能与实践建议

  • ✅ 统一返回结构：封装成通用响应类 Result<T>；
  • ✅ 使用全局异常处理（@ControllerAdvice + @ExceptionHandler）；
  • ✅ 日志与接口分析可配合 AOP；
  • ✅ 接口文档自动生成：使用 Swagger / Knife4j；
  • ✅ Spring Boot 默认全局 JSON 支持（无需手动配置 MessageConverter）。

🔟 一分钟高分总结（可背诵）

@Controller 用于返回视图页面，配合模板引擎使用；
@RestController 是 @Controller 与 @ResponseBody 的组合，用于返回 JSON 数据。
前后端分离项目中一般使用 @RestController，而传统 MVC 项目使用 @Controller。
它们的底层区别在于处理流程不同：
前者通过 HttpMessageConverter 输出数据，后者通过 ViewResolver 渲染视图。
在实际开发中，我会根据模块类型选择不同的注解，以实现高内聚、低耦合的控制层设计。

🎯 面试题：Spring 后端验证注解有哪些？它们的作用是什么？

一、面试官考察点

该题考察候选人是否了解：

  • JSR-303 / JSR-380 Bean Validation 规范；
  • Spring Boot 与 Hibernate Validator 的整合原理；
  • 常见内置注解的使用场景与差异；
  • 如何自定义校验注解与全局异常处理机制。

二、30 秒答题模板（高分开场）

Spring 的后端参数校验基于 JSR-303/JSR-380 标准规范，
其默认实现是 Hibernate Validator。

常用的验证注解包括：

  • 基础约束注解（@NotNull, @NotEmpty, @NotBlank, @Size, @Min, @Max）
  • 格式校验注解（@Email, @Pattern, @Past, @Future）
  • 数值精度控制注解（@DecimalMin, @DecimalMax, @Digits）
  • 复合与自定义注解。

这些注解通常与 @Valid 或 @Validated 结合使用，
并通过 MethodValidationPostProcessor 实现方法级验证。

五、底层实现原理

Bean Validation 是基于 JSR-303 规范 的标准化机制，其实现类为 Hibernate Validator。

底层流程：

  1. Spring 在启动时注册 LocalValidatorFactoryBean；
  2. 当检测到 @Valid 或 @Validated 注解时，会自动调用 Hibernate Validator；
  3. Hibernate Validator 通过反射扫描目标对象字段；
  4. 根据注解类型匹配对应的 ConstraintValidator；
  5. 执行校验逻辑，返回 ConstraintViolation 集合；
  6. 若校验失败则抛出 ConstraintViolationException。

❓ 如何进行分组验证？ 使用 @Validated(GroupA.class) 标注 Controller 方法参数，实体类字段定义 groups = GroupA.class。

在 Spring MVC 中，拦截器（Interceptor） 是一种基于 AOP 思想的机制，
用于在请求到达 Controller 之前、执行之后以及完成视图渲染之后，

拦截器由 HandlerInterceptor 接口定义，常用于：

  • 参数预处理；
  • 性能监控；

四、拦截器与过滤器的区别（面试高频追问）

对比项 拦截器（Interceptor） 过滤器（Filter）
所属框架 Spring MVC Servlet 规范
拦截范围 Controller 层（由 DispatcherServlet 管理） 所有请求（包括静态资源）
实现接口 HandlerInterceptor javax.servlet.Filter
执行时机 DispatcherServlet 调用 Handler 前后 DispatcherServlet 之前
应用场景 登录校验、权限、日志 编码设置、跨域、全局过滤
可访问对象 HandlerMethod（可反射方法信息） 仅 Request / Response
配置方式 Spring 配置类或 XML web.xml 或 FilterRegistrationBean

💡 一般项目中：

  • 全局控制（如 CORS、日志追踪）放在 Filter；
  • 业务相关拦截（如权限、登录态验证）放在 Interceptor。

五、常见面试追问与标准回答

追问 高分回答
❓ preHandle 返回 false 会怎样？ 请求被中断，不会执行 Controller，也不会执行后续拦截器的 preHandle 或 postHandle。
❓ postHandle 与 afterCompletion 区别？ postHandle 在 Controller 执行后但视图渲染前调用；afterCompletion 在整个请求完成（包括渲染后）调用。
❓ 拦截器如何获取 Controller 方法信息？ 通过参数 Object handler，可强转为 HandlerMethod 获取类名和方法名。
❓ 拦截器中如何注入 Service？ 将拦截器声明为 @Component，由 Spring 管理。
❓ 拦截器线程安全吗？ 默认是单例的，不应在成员变量中存放线程不安全的数据。

六、常见陷阱与注意事项

场景 问题 解决方案
静态资源被拦截 默认 /** 拦截所有请求 使用 excludePathPatterns("/static/**") 排除
拦截器顺序混乱 注册顺序不明确 按 registry.addInterceptor() 调用顺序执行
postHandle 不执行 preHandle 返回 false 或 Controller 抛异常 保证放行且无异常
异步请求未执行 afterCompletion 异步线程未触发回调 使用 AsyncHandlerInterceptor 适配

七、高级扩展：Spring 5 新特性拦截器

Spring 5 引入了 AsyncHandlerInterceptor 接口，
它扩展了异步请求的回调，如 afterConcurrentHandlingStarted()，
可用于 WebFlux 场景或异步 Controller。

八、总结（面试背诵版）

Spring MVC 拦截器（Interceptor）是一种基于 AOP 的请求拦截机制，
通过实现 HandlerInterceptor 接口，在请求处理的三个阶段：

  • preHandle()：请求前拦截；
  • postHandle()：Controller 执行后拦截；
  • afterCompletion()：请求完成后拦截。

主要用于权限验证、日志记录、性能监控等业务场景。
执行顺序遵循 preHandle 正序、postHandle & afterCompletion 逆序。
它位于 DispatcherServlet 与 Controller 之间，
是构建高可扩展 Web 框架的重要机制。

🎯 面试题：全局异常处理（Global Exception Handling）

一、面试速答（标准回答）

在 Spring Boot 中，全局异常处理（Global Exception Handling） 是一种基于 AOP 思想的异常捕获机制，
通过使用 @ControllerAdvice 或 @RestControllerAdvice 与 @ExceptionHandler 注解，
可以统一捕获 Controller 层抛出的各种异常并进行集中处理。

其核心目标是实现：

  • 统一异常格式输出；
  • 区分业务异常与系统异常；
  • 提升接口一致性与可维护性；
  • 避免冗余的 try-catch；
  • 防止异常信息泄露。
