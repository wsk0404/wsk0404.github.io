商品服务

支持商品SPU/SKU 管理、价格维护、上架下架、搜索索引同步。
商品搜索功能基于 Elasticsearch 实现多维度筛选。

一、SPU（标准产品单元）

SPU 用于描述产品的“抽象标准模型”，代表一个可以被反复生产、持续销售的产品品类。

在个护行业中，一个 SPU 通常代表：

  • 一个 产品型号
  • 一条 工厂生产线对应的成品类型
  • 一个 统一的功能定义

例如：

  • “负离子恒温吹风机”
  • “声波震动电动牙刷”
  • “三刀头干湿双剃须刀”
  • “纳米喷雾补水仪”

SPU 承载的是 公共属性，包括但不限于：

  • 商品名称
  • 商品类目（如：个护电器/电动牙刷）
  • 品牌（自有品牌）
  • 主功能、产品参数
  • 生产工艺、使用说明
  • 公共图片与主图视频
  • 统一的基础描述

二、SKU（库存单元）

SKU 代表用户最终购买的“具体可售组合”，与库存、价格直接绑定。

SKU 承载了 属性差异 和 库存维度：

典型的个护行业 SKU 属性包括：

  • 颜色：白色、黑色、粉色
  • 功率：1200W / 1600W / 2000W
  • 清洁模式（电动牙刷）：标准/敏感/深洁
  • 刀头数量（剃须刀）：2 头 / 3 头
  • 包装组合：单机版 / 单机+刷头套装 / 礼盒版
  • 插头类型：Type-C / 国标插头

每个 SKU 具备对应的：

  • 唯一 SKU 编号
  • 售价与促销价
  • 独立库存、库存锁定量
  • 重量与体积（用于物流计算）
  • 对应条码

用户下单、库存扣减、秒杀、优惠计算均基于 SKU 进行。

三、SPU 与 SKU 的关联模型

平台通过 SPU → SKU → 属性 → 类目/品牌 的方式构建完整商品树结构。

数据关联示例（个护场景）：

SPU（负离子吹风机）
　│
　├── SKU1（白色 · 1600W · 标准版）
　├── SKU2（白色 · 2000W · 标准版）
　├── SKU3（黑色 · 1600W · 礼盒套装）
　└── SKU4（粉色 · 2000W · Type-C 版）

所有 SKU 共享 SPU 的基础信息，但拥有独立的：

  • 库存
  • 价格
  • 物流参数
  • 促销状态

“悦护云台”采用 SPU/SKU 商品建模体系，基于 SPU 管理个护产品的标准模型，基于 SKU 管理不同颜色、功率、套装、材质等差异属性，并与库存、价格、营销活动深度绑定。该模型不仅满足个护类工厂多规格、多工艺生产的真实业务，也大幅提升库存管理精度、促销灵活性和商品展示能力，为产销一体化提供数据基础。

购物车服务

在“添加购物车”时，购物车服务必须调用商品服务接口，查询并封装完整的 SKU 信息

前端只需传入两个核心参数：

  • skuId：商品的唯一 SKU 标识（与规格属性绑定）
  • count：购买数量

购物车服务会根据 skuId 调用 商品服务的查询接口，
以获取该商品的基础展示与结算信息。

订单服务

完成订单创建、支付确认、发货、取消等全流程。
采用 可靠消息 + 幂等设计，实现订单与库存的最终一致性。

在恒邦衣品电商系统中，订单服务负责生成订单、校验库存、计算价格、扣减优惠券、锁定库存等核心逻辑。
支持两种下单模式：

  1. 直接购买（Buy Now）：
    用户在商品详情页点击“立即购买”，前端传递商品 SKU、数量等信息直接创建订单。
  2. 购物车下单（From Cart）：
    用户从购物车勾选多个商品后，一次性生成合并订单。

两种方式的核心目标一致：生成标准化订单数据结构并触发库存锁定与支付流程。

两种下单方式的区别与流程

对比项 直接购买 购物车下单
来源 商品详情页 购物车模块
前端传参 skuId、count cartItemIds（已选中商品）
商品数据获取 调用商品服务 从 Redis 购物车缓存中获取
价格计算 商品服务实时查询价格 以购物车中保存的价格为基础，再校验
下单逻辑 单商品下单 多商品批量下单
库存处理 单 SKU 锁定 多 SKU 批量锁定
优惠券 可选择性使用 批量计算使用规则

➤ 1️⃣ 直接购买接口

服务调用流程：

  1. 校验用户合法性（用户服务）
  2. 查询商品信息（商品服务：SKU、库存、价格）
  3. 计算订单金额（含优惠券、运费）
  4. 调用库存服务锁定库存
  5. 持久化订单主表与订单项
  6. 发送 MQ 消息（用于库存扣减、延时关单）

➤ 2️⃣ 从购物车下单接口

服务调用流程：

  1. 获取购物车数据（Redis）
  2. 校验商品状态（调用商品服务）
  3. 计算总价与优惠券折扣
  4. 批量锁定库存（库存服务）
  5. 批量保存订单项并生成主订单
  6. 清除购物车中已下单商品

核心技术点

  1. 幂等性控制：使用 Redis 分布式锁防止重复下单
  2. 库存锁定机制：调用库存服务 lockStock()，写入锁定记录（避免超卖）
  3. 订单超时关闭：通过 MQ 延时队列（如 RabbitMQ TTL + DLX）实现订单自动取消与库存解锁
  4. 价格与优惠券一致性：下单前重新校验价格防止恶意修改前端数据
  5. 事务保证：下单过程采用 分布式事务机制（RocketMQ 事务消息 / Seata） 确保一致性

秒杀服务

为大促活动提供高并发下单支持。
通过 Redis 预减库存 + MQ 异步下单 + 限流与令牌机制，实现削峰填谷与抢购公平性。

⚡ 一、功能概述

在恒邦衣品自营电商系统中，秒杀服务用于应对高并发场景下的限时抢购活动（如“双11特卖”、“新品预售”），
系统需保证：

  • 高并发访问不打垮核心系统；
  • 库存扣减与订单生成一致性；
  • 防止超卖与重复下单；
  • 价格、库存、时间的严格限制。

🧩 二、总体设计思路

核心目标：

“削峰、限流、异步下单、最终一致性”。

架构层面：

  1. Redis 缓存库存：活动开始前，将商品秒杀库存与限购信息加载进 Redis。
  2. 请求限流：使用 Redis + Lua 或 Sentinel 限制用户秒杀请求频率。
  3. 库存原子扣减：使用 Redis.decr() 或 Lua 脚本实现原子库存扣减，防止超卖。
  4. 异步下单：请求进入 MQ（消息队列），后台异步生成订单，削峰填谷。
  5. 订单状态校验：通过订单服务+库存服务实现最终一致性。

🚀 六、典型面试简述（可直接背）

我们的秒杀服务采用 Redis + MQ + 异步下单 架构：
秒杀开始前将库存与商品信息加载到 Redis；
用户请求时通过 Lua 脚本进行原子扣减库存，成功后将下单消息异步投递到 MQ；
订单服务异步生成订单并锁定真实库存，实现削峰与最终一致性；
同时通过 Redis 限流、防重与用户标识控制防止重复秒杀与超卖。
整体方案支持高并发秒杀场景，系统稳定性与一致性均有保证。

🎯 七、常见追问与标准回答

面试追问 回答要点
秒杀如何防止超卖？ Redis 原子扣减 + Lua 保证并发安全，再由订单服务最终校验库存。
为什么要用消息队列？ 异步削峰，防止大量请求直接写数据库。
库存一致性如何保证？ 下单失败时 MQ 补偿机制 + Redis 回滚库存。
如何防止重复下单？ Redis Set + 请求幂等 Token 控制。
高并发时如何保证系统不崩？ 限流 + 异步 + 本地缓存 + 降级策略。

优惠券服务

提供优惠券模板、发放、领取、核销功能。
优惠券使用过程采用乐观锁与 Redis 防重机制。

好的王帅锴，我们来详细分析一下优惠券服务在电商工厂自营系统中的实现方式。面试中，面试官通常关心以下几点：领取规则、核销逻辑、与订单结算的结合、库存与幂等控制。下面我整理一个正式面试风格的回答。

🎯 一、功能概述

优惠券服务主要用于：

  • 发放优惠券（用户领取、活动下发、系统赠送）
  • 核销优惠券（订单结算时使用）
  • 查询优惠券状态（可用、已使用、已过期）
  • 管理库存（每个优惠券的发放数量有限）

特点：

  • 与订单服务、用户服务强耦合；
  • 并发量大时需要防止超发或重复使用；
  • 支持活动和全站通用两种使用场景。

🧩 二、主要业务流程

1️⃣ 优惠券发放

  • 用户主动领取：
    1. 前端请求 POST /coupon/claim 并传入 couponId、userId。
    2. 系统校验优惠券是否存在、是否未过期、是否还有剩余库存。
    3. 使用 Redis 原子库存扣减 防止超发。
    4. 将领取信息写入数据库（user_coupon 表）并记录领取时间。
  • 系统/活动发放：
    • 批量生成用户优惠券记录，异步写入数据库和缓存。

2️⃣ 使用优惠券（核销）

  • 用户下单时选择可用优惠券：
    1. 订单服务调用优惠券服务验证有效性（是否属于用户、是否可用、是否在活动期间）。
    2. 计算折扣金额，将优惠券标记为已使用。
    3. 更新 Redis 和数据库状态。
  • 核销策略：
    • 优惠券状态包括：未使用、已使用、已过期
    • 幂等控制：同一订单多次请求只扣一次优惠券。

3️⃣ 查询优惠券

  • 用户端查询可用优惠券：
    • 直接读取 Redis 缓存加快响应；
    • 支持分页、按类型、金额、满减条件筛选。

💡 五、关键技术点

  1. 库存控制：
    • Redis 原子扣减防止超发；
    • 数据库记录作为持久化，定期同步缓存。
  2. 幂等控制：
    • 每个用户领取、核销都进行唯一索引校验；
    • 避免重复扣减。
  3. 高并发处理：
    • 秒杀/限量优惠券使用异步队列处理领取请求；
    • 限流与防刷（Redis + Token 验证）。
  4. 与订单服务集成：
    • 下单时实时校验优惠券可用性；
    • 核销成功后更新订单实际支付金额。
  5. 过期处理：
    • 使用 Redis TTL 或定时任务自动将过期优惠券标记为失效。

🚀 六、面试简述模板（可直接背）

优惠券服务主要负责发放、查询和核销优惠券。
用户领取优惠券时，通过 Redis 原子扣减库存并写入数据库，保证高并发不超发。
下单时订单服务调用优惠券服务进行核销，计算折扣金额并更新状态，使用 Redis + 数据库确保幂等和一致性。
系统支持限量、满减、折扣等多种类型优惠券，并且可以与秒杀、购物车等模块无缝集成。
