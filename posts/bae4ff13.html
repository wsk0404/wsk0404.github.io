🎯 面试题：你在项目中用过 Redis 吗？说说它的使用场景与原理

一、面试速答模板（开场陈述）

是的，我在项目中大量使用过 Redis。
它是一个基于内存的高性能 Key-Value 数据库，常用于实现：

  • 缓存系统（Cache）
  • 分布式锁（Distributed Lock）
  • 消息队列（Pub/Sub、Stream）
  • 会话共享（Session Share）
  • 排行榜、计数器、限流器 等业务场景。

Redis 的核心特点是：
✅ 高性能（内存存储 + 单线程 IO 多路复用）
✅ 高并发（支持 10w+ QPS）
✅ 丰富的数据结构（String、List、Set、ZSet、Hash 等）
✅ 天然支持持久化与分布式集群。

🎯 面试题：Redis 是单进程单线程的吗？

✅ 一、标准回答（开场陈述）

Redis 在早期版本（6.0 之前）确实是单进程单线程模型，
即所有 网络请求、命令执行、数据读写操作 都由一个主线程串行执行。

但从 Redis 6.0 开始，官方为了充分利用多核 CPU，
引入了 I/O 多线程机制 —— 也就是说：

  • 命令执行仍然是单线程的（保证线程安全）；
  • 网络 I/O（读写 socket）是多线程的（提升并发处理能力）。

🧠 二、原理深入讲解

1️⃣ Redis 早期版本（6.0 之前）

  • Redis 是一个单进程单线程模型。
  • 所有操作在一个主线程中执行：
    • 处理客户端请求
    • 解析命令
    • 访问内存
    • 返回结果

📌 依赖事件驱动（IO 多路复用）机制提升并发性能：
使用 epoll（Linux）、kqueue（BSD）、select/poll（通用）实现非阻塞 IO。

⚙️ 三、为什么 Redis 单线程依然很快？

面试官常追问：“单线程不是慢吗？为什么 Redis 这么快？”
✅ 答案可以从 4 个方面分析：

原因 说明
1️⃣ 纯内存操作 所有数据都在内存中，访问速度远超磁盘
2️⃣ 单线程无锁 无需线程切换、锁竞争，避免上下文切换
3️⃣ 高效 IO 模型 使用 epoll + 事件循环机制
4️⃣ 优化的数据结构 内部采用压缩链表、跳表、哈希表等高效结构

🔍 四、单线程模型的影响

优点 缺点
无锁竞争，执行效率高 无法利用多核 CPU 完全性能
顺序执行，线程安全 单个命令执行时间过长会阻塞后续请求（阻塞问题）

扩展问题：
面试官可能会问：

“如果一个慢查询命令阻塞了 Redis 怎么办？”

回答：

  • 使用 异步命令（如 UNLINK、FLUSHDB ASYNC）；
  • 或使用 多实例分片，将不同业务拆分到不同 Redis 进程中；
  • Redis 7.0 开始支持 多线程任务调度（对部分模块进一步并行化）。

🧩 五、总结背诵版（面试 20 秒答法）

Redis 本身是 单进程单线程模型，所有命令的执行在一个线程中完成，
通过 IO 多路复用机制 实现高并发连接处理。
从 Redis 6.0 开始，引入了 多线程 I/O 模型，
网络读写由多个线程并行执行，但命令执行仍保持单线程，
既保证了性能，又保持线程安全。

Redis 之所以快，是因为：

  1. 内存存储
  2. 单线程无锁竞争
  3. 高效 IO 多路复用
  4. 紧凑数据结构与优化算法

🎯 面试题：Redis 如何保证原子性

✅ 一、面试速答模板（适合开场陈述）

Redis 保证原子性的核心在于：

  1. 命令原子性：每个 Redis 命令都是原子执行的；
  2. 单线程模型：Redis 在主线程中串行执行所有命令，没有线程切换与竞争；
  3. 事务原子性：通过 MULTI / EXEC 实现命令队列的打包执行；
  4. Lua 脚本原子性：通过 EVAL 命令在服务器端一次性执行脚本。

简单来说：
Redis 中的“原子性”不是依赖锁或回滚机制，而是通过单线程串行执行来天然实现的。

二、命令级原子性（基础原子性）

Redis 是单线程的，因此：

  • 每个命令在执行时不会被其他命令打断；
  • 命令要么执行成功，要么失败，不会“执行一半中断”。

该命令执行过程：

  3. 写回结果。

由于是单线程串行执行，所以不会有并发修改问题。
这就是 Redis 命令的原子性（Atomic Command Execution）。

🧠 三、事务级原子性（MULTI / EXEC）

Redis 提供了事务机制来保证一组命令的顺序执行：

✳️ 基本语法：

  • DISCARD：取消事务

脚本级 Lua 脚本 (EVAL) ✅ 整个脚本原子执行，性能高

原因 说明
✅ 单线程执行 Redis 使用 Reactor 模式的事件循环，命令串行执行
✅ IO 多路复用 同时监听多个连接，但一次只处理一个命令
✅ 无锁机制 没有并发写入竞争，不存在锁或上下文切换
✅ Lua 执行时阻塞其他请求 脚本执行过程中不会插入其他命令

多客户端同时修改 key 无法原子 用 WATCH + MULTI 实现乐观锁

🧾 九、总结（背诵版）

Redis 的原子性是通过单线程模型天然保证的：

  • 每条命令独立执行，不会被打断；
  • 事务（MULTI/EXEC）将多条命令打包顺序执行；
  • Lua 脚本保证整段逻辑一次性、不可分割地执行；

Redis 不依赖数据库的回滚机制，而是通过串行执行 + 内存操作实现“要么全部执行，要么不执行”的效果。

🎯 面试题：Redis 的集群模式

一、面试速答模板（开场回答）

Redis 集群（Redis Cluster）是 Redis 官方提供的分布式部署方案，
用于解决单机 Redis 的内存容量受限、高可用性不足、无法水平扩展等问题。

目前 Redis 主要有三种集群模式：

  1. 主从复制（Master-Slave）
  2. 哨兵模式（Sentinel）
  3. Cluster 模式（Redis Cluster）

其中：

  • 主从复制解决读写分离；
  • 哨兵模式解决高可用与故障自动转移；
  • Cluster 模式实现真正的分布式存储与水平扩展。

二、单机局限与集群必要性

问题 单机模式的限制
容量限制 Redis 单机受内存限制（例如16GB），数据多会溢出
吞吐量瓶颈 单线程处理请求，性能无法无限扩展
宕机风险 单节点挂掉后服务不可用
不可水平扩展 不能自动分片或扩容

✅ 解决方案：通过“主从 + 哨兵 + Cluster”三种架构逐步演进。

三、Redis 集群模式分类

1️⃣ 主从复制（Master-Slave Replication）

📘 定义：

  • 一个主节点（Master）负责写操作；
  • 多个从节点（Slave）同步主节点数据；
  • 从节点可分担读请求，实现读写分离。

⚙️ 原理简述：

  • 主节点全量复制 → 从节点（RDB dump + 命令传播）；
  • 主节点写操作异步同步给从节点；
  • 主节点宕机后，从节点需人工提升为主节点。

✅ 应用场景：

  • 读多写少；
  • 读性能扩展；
  • 热备份。

⚠️ 缺点：

  • 主节点宕机需手动恢复；
  • 无自动故障转移；
  • 不能解决分片问题。

2️⃣ 哨兵模式（Sentinel）

📘 定义：

Redis Sentinel 是一种 高可用监控机制，
用于自动监控主从架构中主节点状态，并在故障时自动完成主从切换。

🔧 核心组成：

  • Sentinel 进程（监控 + 故障转移 + 通知）；
  • 多个 Sentinel 节点共同选举，防止误判；
  • 投票机制实现主从切换（类似 Raft）。

✅ 功能：

  • 监控（Monitoring）：定期 PING 主节点；
  • 通知（Notification）：主节点异常时广播；
  • 自动故障转移（Failover）：重新选主 + 更新配置。

⚠️ 局限：

  • 数据仍存储在一个主节点；
  • 不支持数据分片；
  • 适合中小型系统的高可用方案。

3️⃣ Cluster 模式（Redis Cluster）🌟（重点）

Redis Cluster 是官方提供的分布式、高可用、自动分片集群方案。

📌 架构组成：

  • 多个 主节点（Master） 分布存储不同 Key 的数据；
  • 每个 Master 至少有一个 Slave 节点；
  • 集群中节点通过 Gossip 协议通信；
  • 自动实现数据分片、路由与故障转移。

⚙️ 数据分片机制（核心原理）

  • Redis Cluster 使用 哈希槽（Hash Slot）分片机制；
  • 共 16384 个槽位（0～16383）；
  • 每个 Master 负责一部分槽；
  • 插入 Key 时，通过 CRC16(key) % 16384 计算槽位。

✅ 示例：

优点：

  • 自动分片（Hash Slot）
  • 节点间自动路由（Redirect）
  • 支持节点动态扩容与迁移（reshard）
  • 高可用（主从 + 故障转移）

⚙️ 节点通信机制（Gossip 协议）

  • 每个节点与部分节点周期性交换状态信息；
  • 传播：节点列表、槽位信息、主从关系；
  • 故障检测：超过一定时间未响应即标记为 fail；
  • 自动 Failover 由其他节点选举新主。

🧠 请求路由机制：

  1. 客户端计算槽位；
  2. 定位到对应 Master；
  3. 若路由错误，节点返回 MOVED 重定向；
  4. 客户端更新本地槽映射表。

⚙️ 故障转移流程：

  1. Master 宕机；
  2. 其 Slave 检测到超时；
  3. 通过投票机制选出新 Master；
  4. 更新槽分配表，广播全网。

🎯 面试题：Redis 的持久化机制是什么？各自的优缺点？开发中一般用哪种方案？

一、面试标准回答（开场思路）

Redis 是一个 内存数据库，为了防止服务器宕机导致数据丢失，它提供了 持久化机制 将数据保存到磁盘中。

Redis 主要有两种持久化方式：

1️⃣ RDB（Redis DataBase）快照持久化
2️⃣ AOF（Append Only File）日志持久化

从 Redis 4.0 开始，还支持 混合持久化（RDB + AOF），结合两者优点。

二、📘 RDB（快照持久化）机制

✅ 原理说明：

RDB 是 Redis 默认的持久化方式。
Redis 会在指定的时间间隔内（例如每隔 60 秒）将内存中的数据快照保存成一个 .rdb 文件（默认 dump.rdb）。

这些快照可用于：

  • Redis 重启后的数据恢复；
  • 主从复制（Slave 启动时加载 RDB）。

⚙️ 触发方式：

  1. 自动触发（配置文件）

  2. 手动触发

    • save：在主线程执行（会阻塞）
    • bgsave：子进程执行快照（非阻塞）

💡 执行流程（bgsave）：

  1. 主进程通过 fork() 创建子进程；
  2. 子进程将内存数据写入 RDB 文件；
  3. 写入完成后通知主进程；
  4. 主进程继续提供服务（无阻塞）。

🧠 优点：

优点 说明
性能好 使用子进程，主线程无阻塞
文件体积小 二进制压缩后较小，适合备份
恢复速度快 RDB 文件加载比 AOF 快
便于迁移 一个文件即可快速部署

⚠️ 缺点：

缺点 说明
数据可能丢失 两次快照之间的数据无法恢复（非实时）
fork 开销大 创建子进程需复制内存页（写时复制机制）
大数据时耗资源 大内存时 RDB 持久化较慢，占用 CPU 和 IO

三、📘 AOF（追加文件持久化）机制

✅ 原理说明：

AOF（Append Only File）通过记录每次写命令来实现持久化。
Redis 在执行写命令时，将命令追加到 AOF 文件（appendonly.aof）中。

重启时，通过重新执行 AOF 文件中的所有命令恢复数据。

四、📘 混合持久化（RDB + AOF）

从 Redis 4.0 开始支持混合持久化（Hybrid Persistence）模式。

✅ 原理：

  • Redis 在重写 AOF 时，会先写入一份 RDB 内容；
  • 然后再将后续增量命令以 AOF 形式追加；
  • 文件中既包含 RDB 的快照，又包含 AOF 的增量日志。

💡 优点：

  • 启动恢复速度快（基于 RDB）
  • 数据丢失极少（基于 AOF）
  • 综合性能较好，是 目前推荐的方案

🎯 面试题：如果 Redis 正在给线上业务提供服务，此时使用 KEYS 指令，会有什么问题？

✅ 一、面试速答模板（标准高分开场）

在生产环境中，直接使用 KEYS pattern 命令是非常危险的。
因为 KEYS 会遍历 Redis 的整个键空间（O(N) 时间复杂度），在 Redis 单线程模型下，它会阻塞主线程，导致线上业务请求延迟显著上升，甚至 Redis 服务卡死。

此时 Redis 不能处理任何其他请求，业务可能出现接口超时、请求堆积甚至服务雪崩。

🧩 二、深入原理分析

🔹 Redis 是单线程模型

Redis 使用单线程事件循环处理所有客户端命令。
任何一个命令的执行时间过长，都会阻塞后续命令的执行。

🔹 KEYS 命令的执行逻辑

当执行：

Redis 的执行步骤：

  1. 进入主线程；
  2. 遍历所有键（全局哈希表）；
  3. 对每个 Key 进行模式匹配（glob-style 匹配）；
  4. 将匹配结果一次性返回客户端。

🔹 时间复杂度与风险

指令 时间复杂度 风险级别 说明
KEYS pattern O(N) 🚨 高风险 遍历所有 Key，阻塞主线程
SCAN O(1) 每次小步扫描 ✅ 安全 非阻塞渐进式扫描

当 Redis 有 1 亿个 Key 时，KEYS * 的执行可能耗时数秒甚至数十秒，期间 Redis 无法处理任何请求。
在高 QPS 线上环境中，这几秒钟的阻塞足以导致雪崩。

🎯 面试题：Redis 与 MySQL 数据如何保持一致性？

✅ 一、面试高分开场陈述（速答版）

Redis 与 MySQL 的数据一致性问题，本质上是一个 缓存一致性问题（Cache Consistency）。

由于 Redis 是内存数据库，常作为 MySQL 的缓存层使用（典型的“Cache + DB”双写架构），
所以必须保证在缓存与数据库更新不同步的情况下，业务数据仍然一致或最终一致。

常见方案包括：

  • 先删缓存后更新数据库；
  • 先更新数据库再删缓存；
  • 延迟双删策略；
  • 异步消息队列同步；
  • Canal / Binlog 实时订阅同步。

🧩 二、问题背景

典型架构：

常见问题：

  1. 更新了 MySQL，但 Redis 仍然是旧数据；
  2. 更新了 Redis，但 MySQL 未更新成功；
  3. 并发情况下数据脏读；
  4. 缓存失效引发雪崩或穿透。

⚙️ 三、三种缓存写入模式（重点）

模式 操作顺序 特点 一致性风险
Cache Aside（旁路缓存） 读：先读缓存 → 缓存未命中再查库并写缓存；写：更新数据库后删除缓存 最常见策略 ✅最终一致性好，⚠️短暂不一致
Read/Write Through 应用只访问缓存，由缓存负责读写数据库 一致性较好 实现复杂
Write Behind（异步写回） 数据先写缓存，异步同步到数据库 性能高 ⚠️丢失风险大，不保证强一致

✅ 绝大多数系统使用 Cache Aside 模式（旁路缓存），配合“更新数据库 + 删除缓存”策略。

🔍 四、两种主流一致性维护策略

（1）❌ 方案一：先删除缓存，再更新数据库

问题：
如果第二步失败（数据库未更新），则缓存被删除但数据库仍是旧值。
下次读请求重新回填旧数据 → 数据不一致。

❌ 不推荐。

（2）✅ 方案二：先更新数据库，再删除缓存

优点：

  • 即使删除缓存失败，也可通过 TTL 或延迟重试修复；
  • 能保证最终一致性。

问题场景：
若两个线程同时操作：

➡ 导致缓存中再次出现旧数据（短暂不一致）。

🧠 五、解决缓存与数据库不一致的“延迟双删策略”

⭐ 生产最常用方案：更新数据库 + 延迟删除缓存

核心思想：

  • 第一次删除缓存避免读旧数据；
  • 延迟二次删除确保在并发场景下再次清理被回写的旧缓存。

改进做法：
使用异步任务或消息队列延迟二次删除（非阻塞主线程）。

🔄 六、异步一致性方案（MQ / Canal）

✅ 方案一：基于消息队列同步

  • 写数据库成功后发送更新事件到 MQ；
  • 消费端异步删除或更新缓存；
  • 可配合重试机制保证最终一致性。

优点：

  • 异步解耦；
  • 可靠性高，可重试；
  • 延迟低。

✅ 方案二：基于 Canal / Binlog 同步机制

  • 通过订阅 MySQL Binlog 实时监听数据变化；
  • 一旦检测到更新/删除操作，自动清理对应 Redis 缓存。

优点：

  • 与业务逻辑解耦；
  • 一致性强；
  • 实时同步。

常用于： 大型分布式缓存同步场景（如商品详情页、库存系统）。

🧩 七、缓存更新的最终一致性设计要点

问题 解决思路
缓存失效瞬间并发查询导致穿透 加分布式锁或本地互斥锁（如 Redisson Lock）
缓存删除失败 加异步补偿任务（MQ / 定时任务）
热点 Key 高频更新 采用延迟队列批量更新
异步更新延迟 通过 Canal 实时订阅数据库变更
高一致性要求场景（如支付） 放弃缓存，直查数据库

🚨 二、缓存穿透（Cache Penetration）

📘 定义：

指 查询一个在缓存和数据库中都不存在的 key。
由于缓存未命中，每次请求都要落到数据库，造成数据库压力暴增。

📌 举例：

用户请求了一个根本不存在的用户 ID（例如负数或无效 ID），
每次都绕过 Redis 查 MySQL。

⚙️ 解决方案：

方案 原理 适用场景
✅ 缓存空对象 将查不到的结果缓存为空值（如 null）并设置较短 TTL，防止重复查询。 常规业务场景
✅ 布隆过滤器（BloomFilter） 在查询缓存前，用布隆过滤器判断 key 是否可能存在。 大数据量高并发系统
✅ 参数校验 对非法参数提前拦截（如 ID<0）。 网关层优化

⚡ 三、缓存击穿（Cache Breakdown）

📘 定义：

某个 热点 Key 在缓存过期的瞬间，有大量并发请求同时访问，

🌪️ 四、缓存雪崩（Cache Avalanche）

📘 定义：

指 大量缓存同时失效 或 Redis 服务宕机，
所有请求直接访问数据库，造成数据库雪崩式崩溃。

📌 举例：

午夜缓存批量设置相同 TTL（如 1 小时），同时过期导致数据库瞬间过载。

⚙️ 解决方案：

方案 原理 效果
✅ 随机 TTL（过期时间加随机数） 避免同一时间大量 Key 同时过期。 缓解过期集中问题
✅ 多级缓存（如本地 + Redis） 提高缓存层级，Redis 崩溃时可短暂兜底。 高可用
✅ 服务降级 / 熔断机制 Redis 不可用时快速失败或使用兜底数据。 防止数据库雪崩
✅ Redis Cluster + 持久化 提升 Redis 可用性，防止整体宕机。 架构层优化

🔁 五、缓存一致性问题（Cache Consistency）

📘 定义：

缓存与数据库内容不一致，导致用户读取到旧数据或脏数据。

📌 原因：

写操作时未同步更新缓存或延迟更新。

⚙️ 解决方案（见前题详解）：

  1. ✅ 采用 Cache Aside（旁路缓存） 模式；
  2. ✅ 先更新数据库，再删除缓存；
  3. ✅ 延迟双删策略；
  4. ✅ 消息队列异步同步；
  5. ✅ Canal 监听 Binlog 实时清理缓存。

🔥 六、热点 Key 问题（Hot Key）

📘 定义：

单个或少数 Key 被极高频访问，导致 Redis 单线程处理瓶颈或热点节点过载。

📌 举例：

“双十一秒杀商品ID=1001”的缓存，每秒上万请求访问。

⚙️ 解决方案：

方案 说明
✅ 本地缓存 + Redis 双层缓存 使用 Caffeine 或 Guava Cache 缓解热点压力。
✅ 热点 Key 预分片 将一个 Key 拆成多个（如 key_1, key_2）。
✅ 热点数据异步刷新 定时任务提前预热缓存。
✅ 限流 + 熔断 防止瞬间高并发击穿缓存层。

✅ 一、面试高分开场陈述（速答版）

简单来说，就是提前把常用的数据加载进缓存，让缓存一开始就是热的，

🎯 目标：

  • 减轻数据库压力；
  • 提升首屏响应速度；

🎯 面试题：Redis 分布式锁（Distributed Lock）

✅ 一、标准开场回答（面试高分版）

Redis 分布式锁是一种利用 Redis 的原子性操作（SETNX、EXPIRE、SET EX NX PX 等）实现的分布式并发控制机制，
用于在分布式系统中保证同一时刻只有一个节点（线程）能获得资源操作权限。

它的核心目标是：
防止多个节点同时修改共享数据，保证分布式环境下的互斥性与一致性。

🧩 二、为什么需要分布式锁？

在单体应用中，我们通常使用：

  • synchronized
  • ReentrantLock

但在分布式系统中：

  • 服务部署在多台机器；
  • 多线程同时访问同一个共享资源（如库存、订单号、任务执行标志）；
  • JVM 级锁已无法生效（不同进程内存空间隔离）；

此时必须使用分布式锁，如 Redis 或 Zookeeper 实现的锁机制。

🧭 六、RedLock 算法（多节点一致性锁）

为了防止 Redis 主从切换导致锁丢失，Redis 作者 Antirez 提出 RedLock 算法。

🌍 核心思想：

在多个独立 Redis 实例上同时尝试加锁，
若超过半数（n/2+1）实例加锁成功，则认为锁获取成功。

步骤：

  1. 获取当前时间戳；
  2. 依次向 N 个 Redis 节点请求加锁（超时限制）；
  3. 若超过一半节点加锁成功 → 获取成功；
  4. 若失败 → 释放所有已加的锁；
  5. 设置锁有效期防止死锁。

⚠️ 缺点：实现复杂、网络延迟风险大。
➡️ 实际生产中常用 Redisson 替代手写 RedLock。

🎯 面试题：Redis 布隆过滤器（Bloom Filter 原理与应用）

一、面试速答模板（高分开场）

布隆过滤器（Bloom Filter）是一种高效的概率型数据结构，
用于判断一个元素是否存在于集合中。

它的特点是：

  • 判断为不存在 → 一定不存在；
  • 判断为存在 → 可能不存在（存在误判）。

在 Redis 或分布式缓存系统中，布隆过滤器常用于：
✅ 防止缓存穿透
✅ 高效地进行大数据集合的存在性判断
✅ 节省内存空间

二、布隆过滤器的核心原理

🔹1️⃣ 数据结构

布隆过滤器由两部分组成：

  1. 一个长度为 m 的二进制位数组（BitMap）
  2. k 个独立的哈希函数

🔹2️⃣ 元素插入过程

当我们插入一个元素 x 时：

1️⃣ 通过 k 个哈希函数计算出 k 个哈希值
2️⃣ 将这 k 个位置的比特位设为 1

🔹3️⃣ 元素查询过程

当判断一个元素是否存在时：

1️⃣ 计算该元素的 k 个哈希值
2️⃣ 检查对应比特位是否全为 1：

  • 如果有任意一个为 0 → 一定不存在；
  • 如果全为 1 → 可能存在（存在误判的可能性）。

三、布隆过滤器的优缺点

类型 说明
✅ 优点 占用空间极小、查询速度快（O(k) 时间复杂度）
✅ 优点 插入和查询效率都很高，适合大数据场景
⚠️ 缺点 存在一定误判率（可能存在，但实际上不存在）
⚠️ 缺点 不支持删除操作（标准布隆过滤器）

场景 说明

七、误判率计算与优化

布隆过滤器的误判率 P 由如下公式决定：

[
P = \left(1 - e^{-\frac{k \times n}{m}}\right)^k
]

  • n：插入的元素数量
  • m：BitMap 位数
  • k：哈希函数数量

结论：

  • m 越大，误判率越低；
  • k 合理取值（通常 6~10）可平衡空间与性能。

面试官提问 高分回答
❓ 为什么说布隆过滤器有误判？ 因为不同元素可能通过哈希碰撞映射到相同 bit 位，从而导致“假阳性”。
❓ 怎么避免误判？ 无法完全避免，但可通过调大 bit 数组（m）和合理设置 k 减少概率。
❓ 支持删除吗？ 标准 Bloom Filter 不支持，可用 Counting Bloom Filter（计数型）实现删除功能。
❓ Redis 中如何使用？ 使用 RedisBloom 模块（BF.ADD、BF.EXISTS 命令），或者用 BitMap + Hash 自行实现。
❓ 布隆过滤器与 Set 区别？ Set 无误判但占内存大；布隆过滤器节省内存但存在误判，适合海量场景。

九、实际开发案例：防止缓存穿透

✅ 问题背景

用户频繁访问不存在的 key，导致请求直接打到数据库。

✅ 解决方案

  1. 将数据库中所有合法 key（如用户 ID）提前加入布隆过滤器；
  2. 每次访问前先判断：
    • 不存在 → 直接返回；
    • 可能存在 → 查询 Redis 缓存；
    • 缓存无数据 → 访问数据库并更新缓存。

🔟 总结（背诵版）

布隆过滤器（Bloom Filter）是一种基于BitMap + 多哈希函数的空间高效数据结构，
可用于快速判断元素是否“可能存在”于集合中。
它的优势是高性能、低内存占用，但存在误判、不支持删除的限制。

在 Redis 中，布隆过滤器常用于防止缓存穿透、URL 去重、黑名单过滤等场景，
可通过 RedisBloom 模块实现生产级使用。
